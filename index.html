Starter Upper: Setup git and project hosting quickly

In this course, we'll use git repositories and project hosts (such as Github and Gitlab) to submit work, receive assignments and feedback. Git repositories store snapshots of a folder over time, enabling collaborators to work simultaneously along different time-lines called branches, which can be merged together. Project hosts manage access to git repositories over the network.

Initially, your repository will be a copy of the course repository. To submit work, first add files to git, then commit to record a snapshot, and finally push to your repository. The instructor will fetch and checkout your work to review it. The instructor may also update the course repository; you may pull to receive those updates.

Hints for using git
  • Don't push what's broken: test beforehand
  • Use branches for work-in-progress
  • Commit often, and to complete a merge
  • Do not push to upstream
  • Use a merge tool if you see a CONFLICT

Receive updates following these steps 
cd ~/REPOSITORY
git commit -am "Message goes here"
git fetch --all
git merge upstream/master
git mergetool

Local setup: Setup git on your machine

Git needs to know your name, email address, and where to find the repositories we created. Git uses your name and email address to identify the author of each snapshot (commit). Git also keeps a list of remote repositories in your local repository, so that you can push to your private repository to submit work and also receive updates from upstream by doing a fetch then a merge or a pull. The command git remote -v shows the nickname and URL of every remote repository:

  • The course repository (upstream)
  • Your private repository (bitbucket)
  • Your private repository (github)
  • Your private repository (gitlab)

  • DO NOT move your local git repository into any folder synchronized with services such as Dropbox or Google Drive
